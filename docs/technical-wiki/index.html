Isomer Next codebase guide

Isomer Next is a pnpm/Turborepo monorepo for a government CMS and site builder. Studio manages content and publishing workflows; shared packages render published sites, own the database schema, and provide internal runtime utilities.

Architecture at a glance

System map

Studio is the editing control plane. Published sites are static Next.js outputs built from database snapshots and rendered by the component package.

1

Editor signs in

Email OTP and optional Singpass flows create an iron-session.

2

Studio loads site

Pages call feature modules, which use tRPC React Query hooks.

3

Routers call services

tRPC procedures enforce auth, rate limits, logging, and input schemas.

4

Services mutate data

Kysely/Prisma write resources, blobs, versions, permissions, and audit logs.

5

Publish triggers build

Content publish creates a Version; site deploy triggers AWS CodeBuild.

6

Template renders static site

Build scripts export JSON and the template renders with isomer-components.

  -> Next.js pages in apps/studio/src/pages
  -> Feature modules in apps/studio/src/features
  -> tRPC client in apps/studio/src/utils/trpc.ts

Primary application

Studio app architecture

apps/studio is a Next.js 16 application using the Pages Router. Pages stay thin and delegate UI to vertical feature modules.

Entry points

  • src/pages/_app.tsx wraps providers.
  • src/pages/api/trpc/[trpc].ts exposes tRPC.
  • src/server/context.ts creates request context.
  • src/instrumentation.ts starts tracing and jobs.

UI and state

  • Chakra UI plus OGP design system for Studio.
  • React Query through tRPC hooks for server state.
  • Jotai and React Context for local editor UI state.
  • JSONForms and TipTap power the editing experience.

Server surface

  • Routers are defined in src/server/modules/*.
  • Services hold business logic and permission checks.
  • Kysely is preferred for new database work.
  • Prisma still supports legacy and generated-client paths.

Runtime path

Request lifecycle

  1. Route and layout

    A file under apps/studio/src/pages maps to a route. Most pages use a getLayout pattern and delegate content to a feature component.

  2. Feature module calls tRPC

    Feature components call trpc.*.useQuery, useMutation, or suspense hooks from src/utils/trpc.ts.

  3. API handler invokes app router

    pages/api/trpc/[trpc].ts passes requests into appRouter, with SuperJSON serialization and batching controls configured in the tRPC handler/client.

  4. Middleware applies cross-cutting concerns

    src/server/trpc.ts applies GrowthBook cleanup, request logging, app-version headers, content-type checks, auth, webhook keys, and rate limits.

  5. Router delegates to service

    Routers wire validated input to service functions. Services perform authorization and database work, often inside a transaction with audit logging.

Edit to deploy

Content and publishing flow

Content contract

Page content is JSON shaped by @opengovsg/isomer-components schemas. Studio uses JSONForms and AJV validation to edit and persist blobs. 

{
  "version": "0.1.0",
  "layout": "content",
  "page": { "...": "layout-specific metadata" },
  "content": [{ "type": "paragraph", "...": "block props" }]
}

Publishing semantics

  • Draft saves update Resource.draftBlobId.
  • Content publish copies draft JSON into a new Version.
  • Resource.publishedVersionId points at the live snapshot.
  • Site deploy triggers AWS CodeBuild when the site has a launch project.
  • Scheduled publish is handled by a pg-boss cron job every minute.

Published-site build pipeline

  1. Export: tooling/build/scripts/publishing/index.ts queries the DB for site content and writes JSON artifacts.
  2. Prepare: tooling/build/scripts/publisher.sh assembles the template and exported data.
  3. Render: tooling/template loads JSON and uses RenderEngine.
  4. Deploy: output is synced to S3 and CloudFront origin paths/redirect objects are updated.

Access control

Authentication and permissions

Authentication flow

  • Session storage uses iron-session.
  • Email OTP lives under server/modules/auth/email.
  • Singpass OIDC lives under server/modules/auth/singpass.
  • Soft-deleted users are rejected by authMiddleware.
  • Whitelist checks gate access after user lookup.

Authorization model

  • CASL abilities map roles to resource/site/user-management actions.
  • Roles are Editor, Publisher, and Admin.
  • Site-level permissions are the common case today.
  • Isomer admins can receive admin-equivalent overrides.
  • Permission checks belong in service methods before mutation.

Files and specialized publishing

Assets and gazettes

Assets

Assets are not database rows. They are stored in S3 under keys scoped by site and referenced by content JSON.

  • asset.router.ts exposes upload endpoints.
  • asset.service.ts issues presigned PUT URLs.
  • SVG upload is server-side sanitized before storing.
  • Soft delete uses S3 object tags such as deletedAt.

Gazettes

e-Gazette is a specialized collection/PDF workflow with a separate bucket and SearchSG document-ingestion job.

  • gazette.router.ts and gazette.service.ts own the workflow.
  • Access is constrained to Toppan email domains or Core/Migrator admins.
  • PushDocumentJob tracks SearchSG indexing.

Published-site renderer

@opengovsg/isomer-components

This package renders published Isomer sites. It owns templates, page layouts, block components, JSON Schema, interfaces, render engine utilities, Tailwind presets, and Storybook.

Public exports

  • . exports engine, schemas, types, interfaces, presets, hooks, constants.
  • ./templates/next exports render helpers for Next template usage.
  • Package output is ESM under dist/esm.

Template structure

  • templates/next/components/native: atomic content blocks.
  • templates/next/components/complex: composed blocks.
  • templates/next/layouts: page-shape templates.
  • templates/next/render: layout/block dispatch.

How to change the system safely

Conventions

Studio conventions

  • Pages are thin; real UI lives in feature modules.
  • Feature modules should not cross-import each other.
  • Routers should stay thin; services own business logic.
  • Input validation belongs in apps/studio/src/schemas.
  • Permission checks and audit logging must happen in service-layer mutations.
  • Use useZodForm for forms when applicable.

Package conventions

  • Prefer workspace/catalog dependency specifiers over raw versions.
  • Components render; Studio produces content.
  • Schema additions in components must be backward compatible.
  • Use ~/lib/tv and ~/lib/twMerge wrappers in components.
  • New public components need Storybook coverage.
  • Kysely is preferred for new DB queries.

Risks and sharp edges

RiskWhy it mattersMitigation
Client login flagLocalStorage only drives UX redirects and is bypassable.Never treat it as auth; use protected tRPC procedures.
Dual ORMKysely and Prisma coexist and can produce inconsistent patterns.Prefer one transaction owner per mutation; new code should favor Kysely.
Schema compatibilityComponents schema changes can affect every published site.Add optional fields first; coordinate Studio producer changes separately.
Audit log integrityMutations without audit trail are operationally risky.Log real DB rows in the same transaction as the write.
External-service callsEmail, S3, Singpass, SearchSG, CodeBuild can be abused or fail.Use procedure rate limits, generic errors, and structured logging.
Build type safetyNext build can ignore TypeScript build errors.Run pnpm typecheck in CI and before risky merges.
Generated artifactsDB clients and generated Kysely types can drift.Run pnpm generate after schema changes.
